1. 系统边界
选择解决的问题

接收业务系统提交的通知请求，立即持久化，解耦投递过程
按供应商配置（URL、Header、Body）路由并发送 HTTP 请求
失败重试与退避策略，保障最终投递成功
永久失败后的告警与人工干预入口
内部调用方身份校验（防止未授权写入）

明确不解决的问题
不解决的问题原因外部 API 的响应值处理业务方明确不需要；处理响应会引入额外耦合幂等性保证（供应商侧去重）无法控制外部系统；由供应商自行保证或业务方提供幂等 key请求体的数据转换 / 映射属于业务语义，不应放在通用投递层供应商 API 的限流适配各供应商策略差异大，v1 不做，超时/失败走重试即可通知顺序保证要求严格顺序则需大量牺牲吞吐，本场景无此要求


2. 可靠性与失败处理

投递语义：至少一次（At-Least-Once）
选择 at-least-once 而非 exactly-once，原因如下：

Exactly-once 需要外部系统配合分布式事务或幂等 token，我们无法控制供应商
At-least-once 通过数据库持久化 + 重试，实现简单、可靠
业务方（如有需要）可在请求中携带 idempotency_key，由供应商侧去重

接入层：快速落盘，解耦投递
业务系统调用我们的接口后，只做两件事：写库 + 返回 202 Accepted。不等待实际 HTTP 调用结果，消除了对外部可用性的依赖，接口 P99 稳定在毫秒级。
重试策略：指数退避 + 抖动
delay(n) = min(2^n, 480) 分钟 + random(0, 60) 秒
第 n 次失败下次重试时间1~1 分钟后2~2 分钟后3~4 分钟后5~32 分钟后8~4 小时后10（上限）进入 DEAD
抖动（jitter）的作用是防止大量失败任务在同一时刻集中重试，形成惊群冲击外部系统。
外部系统长期不可用的处理
当外部系统出现持续性故障时，任务会在退避队列中自然积压，不会丢失：

短暂故障（< 数小时）：靠退避重试自动恢复，无需人工介入
长期故障（> 最大重试窗口）：任务进入 DEAD 状态并触发告警，运维人员可在故障修复后通过管理接口批量重放 DEAD 状态记录
供应商彻底废弃：管理员标记该供应商为 disabled，DEAD 队列中的记录保留审计，不再重发
