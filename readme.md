问题的理解：
    这本质上是一个异步可靠投递问题，而不只是"HTTP 转发代理"。业务系统触发事件（如用户注册、支付成功）时，需要通知外部供应商，但：
    业务系统不关心结果，不想等待
    外部供应商 API 可能挂掉、超时、限流
    各家供应商接口格式不统一
    如果直接在业务系统里同步调用外部 API，任何一个供应商的抖动都会影响核心业务链路。这是这个系统存在的根本原因。

整体架构与核心设计
    整体思路：用数据库做可靠队列，MVP 优先不引入 Kafka/RabbitMQ。原因很简单：消息队列解决的是高吞吐 + 多消费者的问题

关键工程决策与取舍说明：
    1. At-Least-Once 而非 Exactly-Once
    放弃精确一次投递，原因很直接：exactly-once 需要外部供应商配合（幂等接口或分布式事务），而我们无法控制外部系统。选择 at-least-once 配合数据库持久化+重试，把复杂性留在供应商侧，自身实现保持简单可靠。代价是极少数情况下可能重复投递，这在本场景是可接受的。
    2. 接入层只写库，不等投递结果
    业务系统调用接口后立即返回 202，实际 HTTP 投递完全异步。这一决策把我们自身的可用性与外部系统的可用性彻底解耦——哪怕目标 API 宕机数小时，接入接口的 P99 依然稳定在毫秒级。代价是引入了额外的存储和 Worker 层复杂度（以及后来为解决事务一致性又引入的 Outbox Pattern），但这个复杂度是值得的，因为它把不确定性从关键路径上移除了。
